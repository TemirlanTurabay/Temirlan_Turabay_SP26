7. How to evaluate whether the current setup is sufficient or risky

The current setup has some good security features. The database uses scram-sha-256 for password encryption, which is a modern and safer method. Also, most system roles do not have login access, and there are no extra custom users with broad privileges shown in the output.

At the same time, there are several risks. The database is being used through the postgres superuser, which has full access and can bypass all restrictions. In addition, row-level security is not enabled on any table, and there are no RLS policies in the public schema. The server also has ssl = off, listen_addresses = '*', and the database connection limit is unlimited, which can make access less secure if the server is exposed to a network.

Overall, the protection is only partially sufficient. Basic privilege management exists, but important protections for customer data are still missing.


8. Example of at least one vulnerability and an improvement

One important vulnerability is the absence of row-level security on customer-related tables such as rental and payment. Because of this, if a user gets SELECT permission on these tables, that user could see records of all customers, not just their own.

This is a privacy and confidentiality risk because rental history and payment history are personal data. A good improvement is to enable row-level security on these tables and create policies so each customer role can view only its own rows.

Another weakness is that SSL is turned off, while the server listens on all interfaces. This means data may travel without encryption if remote connections are allowed. A better solution is to enable SSL and restrict network access only to trusted hosts.


9. Ready-to-use conclusion text

Based on the audit, the dvdrental database already uses PostgreSQL roles, privileges, and secure password encryption. These settings provide basic protection at the object level.

However, the configuration is not fully sufficient because customer data is not protected by row-level security. The main problem is that the database relies too much on broad administrative access and does not limit users to only their own records.

To improve security, it is recommended to enable row-level security on rental and payment, avoid using the postgres superuser for normal work, and enable SSL for safer connections.


10. Short final summary example

The database already has basic security mechanisms such as roles, privileges, and SCRAM-SHA-256 password encryption. But this is not enough for full protection because row-level security is missing, SSL is disabled, and the server can listen on all network interfaces.

A practical improvement is to enable RLS for customer tables, use separate limited roles instead of the postgres superuser, and turn on SSL. This will better protect customer data and reduce the risk of unauthorized access.
